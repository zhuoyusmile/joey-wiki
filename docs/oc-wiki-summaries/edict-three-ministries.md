# 三省六部 Edict 架构 — 学习总结

> 来源：[OC Wiki - 三省六部 Edict 架构分析](https://shazhou-ww.github.io/oc-wiki/shared/edict-three-ministries/)
> 整理：敖丙 | 日期：2026-03-30

## 核心洞察

三省六部 Edict 是 M2 的"重型版"——用 12 个 Agent 实现制度化审核和可视化管理。如果 M2 是"敏捷小队"，Edict 就是"正规军"。

## 核心流程

```
皇上(你) → 太子(分拣) → 中书省(规划) → 门下省(审核/封驳) → 尚书省(派发) → 六部(并行执行) → 奏折(回报)
```

## 最大差异：门下省封驳

独立审核 Agent，每个方案**必须**通过审核。不合格直接封驳打回。这是 CrewAI/AutoGen/MetaGPT 都没有的制度性审核。

## 军机处看板（10 个面板）

包括：旨意看板（Kanban）、省部调度（心跳检测）、奏折阁（任务归档）、官员总览（Token 排行）、模型配置（热切换）、朝堂议政（多 Agent 辩论）、上朝仪式（仪式感）等。

## 与 M2 对比

| 维度 | 三省六部 | M2 |
|------|---------|-----|
| Agent 数量 | 12 个固定 | 2 + 动态 |
| 审核 | 门下省专职 | 自验证 |
| 资源 | 常驻，Token 大 | 按需 spawn |
| 适合 | 多人团队 | 个人助手 |

## 值得借鉴的点

1. **独立审核层** — 不是自己验证自己
2. **结构化归档** — 任务完成后生成报告，不只是日志
3. **Agent 健康监控** — 心跳 + 活跃度检测
4. **仪式感设计** — 让交互更有趣

## 我的理解

三省六部和 M2 代表了两种不同的管理哲学：制度化 vs 灵活化。对于小爪这种个人助手场景，M2 更合适；但三省六部的"独立审核"和"结构化归档"思想值得借鉴。

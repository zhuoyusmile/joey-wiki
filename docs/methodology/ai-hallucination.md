## 楔子：我们到底在防什么

跟 AI 干活几个月之后，越来越确认一件事——**单 AI 的「我觉得没问题」，几乎不能作为任何决策依据**。它不是骗你，它是真的相信。模型对自己的产出有一种**结构性自信**：写错了它会自圆其说，编造了引用它会贴出格式工整的来源，跑测试它会告诉你「应该过了」。

这套方法论不是教你怎么把 AI 用得更花，是教你怎么**默认不信它的自我评价**——靠跨家族独立、真证据、机制化打分把幻觉钉在台面上。下面每一章都从踩过的坑里提炼，结尾给一句话总结。

## 一、AI 幻觉的几种典型形态

不是所有幻觉都是「编个不存在的 API」。实战里更难抓的是下面这几种：

| 形态 | 特征 | 实战例子 |
|---|---|---|
| **写错事实** | 朝代、人名、引文搞混 | writer agent 把"靖康"年间的官职混进了元丰改制前的称谓 |
| **编造引用** | 引一段「文档说」其实没说 | chrome cookie API 行为，前两轮 fix 都是 Opus 4.6 凭"文档+mock"推断，全错 |
| **padding 凑字数** | 同段复制 N 次或硬塞排比凑长 | Gemini 3.x 长输出会重复段落 4 次冒充 363 行 |
| **placebo 测试** | 测试硬编码字符串、不读真文件 | 单测全绿但真机跑炸 |
| **AI 味套话** | 章尾"风+马声+闭眼"三件套、"不是 X，是 Y"句式、作者代人物总结 | 《靖康》卷二 9 章犯 8 次同一种章尾节拍器 |
| **自我感觉良好型评分通胀** | reviewer 越审分越高，标尺漂移 | writer agent 修订纵向追，分数从 78 飘到 88，横向校准发现"新版甚至不如老版" |

**硬规矩**：把这 6 类列成清单贴到所有 reviewer 的 prompt 里，让它们**显式**回答"这次有没有踩到第 N 类"。让幻觉从背景噪音变成可勾选项。

## 二、多 agent 调度的核心原则

小虎那句话已经定了调：**多 AI 的价值不在「人数」，在「独立性」**。两个同 family 模型互审 = 一个人左手审右手，盲点高度重叠，给你 2× APPROVE 也只是 1× 信号。

固化下来的几条原则：

1. **跨家族独立**：reviewer 必须是不同模型家族（Claude × GPT × Gemini）。两个 Opus 互审不算双审。详见第三章。
2. **实施者 ≠ reviewer**：写代码的 agent 不审自己的代码，写章节的 writer 不审自己的章节。曾连续 6 轮派"实施者 + GPT-5.5"，GPT 不可用那几轮就退化成自审 + 单审，被抓后 retroactively 补审。
3. **协调者必自验**：reviewer 给完意见、实施者改完汇报"已修"，主 agent **必须自己再跑一遍** `git diff --stat` / `npm test 2>&1 | tail -5` / 抽样查"Padding to reach 3KB"这种凑字数痕迹。不是"试探性从宽"，是硬红线。
4. **求教姿态**：派 reviewer 时永远是"请你重新看一眼"、"该坚持就坚持"，绝不是"以 Joey 为准"、"母题保护令"这种以上压下的口吃。Joey 的反馈很重："**我不是高高在上的皇帝，我只是向夫子求教的小学生**"。命令式 prompt 会让 reviewer 反考设装、改判迎合，反而毁掉独立性。

**实战翻车**：
- 早期派 reviewer A 时说"重点审商业流畅感"，派 reviewer B 时说"重点审文学性"——视野提前框住，框架外的硬伤全漏。
- 派 reviewer 用"Joey 否决了上一轮 reviewer 的意见"措辞——下一轮 reviewer 直接放水。

**硬规矩**：
- 派 reviewer 默认**全方位、不预设角度**，prompt 里**明写"不预设审核角度"**，让对方知道是有意的不是忘了。
- 只有 Joey 明确说"这次只看 X"才预设角度。
- 双审两位**并发跑**，互不可见对方输出。

## 三、多模型审核：1 写 + 2 审 标准配方（核心）

这一章是整套方法论的承重墙。其它章节里凡是涉及"派 reviewer"的，默认按本章配方走。

### 3.1 为什么必须多模型审

**单 AI 自信 = 单点偏差**。同一个模型（甚至同一个 family）有共享的训练分布、共享的语言习惯、共享的判分倾向——它觉得"这段挺好"，是因为它自己写出来也长这样。让它审自己同 family 的兄弟，就像让一个人评价自己的口音，几乎不可能听出问题。

**跨家族能锤穿同 family 共同盲点**。Claude 系（Opus / Sonnet）对结构和语言节奏敏感、对"AI 味"也最容易共谋；GPT 系对边界条件、事实校验、安全漏洞嗅觉强；Gemini 系长上下文便宜、能整段扫描重复 padding。三家放在一起，A 看不出来的 B 看出来，B 错过的 C 抓回来——同 family 的"集体幻觉"才有机会被外部坐标系刺穿。

小虎在 multi-ai-review.md 里已经把这一点论证到底：单 AI review 是"自带答案的考试"；跨家族 review 才是"换一张卷子重新做"。

### 3.2 标准配方：1 写 + 2 审，并发派出

```
implementer  ×1   ──写──▶  artifact
                              │
                              ├──▶ reviewer A  (家族 X)  ┐
                              └──▶ reviewer B  (家族 Y)  ┴── 并发
                                                          │
                                              两份独立报告 → 协调者 agent
```

**关键约束**：
- **1 个 implementer**：1 个 agent 全权负责写、改、跑测试，不要"两个写手协作"——会扯皮 + 责任真空。
- **2 个 reviewer**：跨至少两个家族，一定要**并发派出**（同时下发任务，互不可见对方输出）。
- **并发优势**：一是省时间（一份审 30 分钟，串行 60 分钟，并行 30 分钟）；二是**两份独立报告比一份深入报告更能暴露问题**——独立性是双审唯一的价值，串行 / 互见会污染独立性。

### 3.3 跨家族选型清单

| 家族 | 可用模型 | 强项 |
|---|---|---|
| Claude | Opus 4.7 / Opus 4.6 / Opus 4.6 1M / Sonnet 4.6 | 长文结构、语言节奏、文学性、长上下文（4.6 1M） |
| GPT | GPT-5.5 / GPT-5.4 | 边界条件、事实核查、真机 repro、安全 |
| Gemini | Gemini 3.1 Pro | 超长上下文便宜、整段重复 / padding 嗅觉 |

**选型铁律**：review 必须**至少跨两家**。不允许"Opus + Sonnet"凑双审（同 family 不算）。常见组合：
- 代码任务：implementer = Opus 4.6 1M，reviewer = GPT-5.5 + Gemini 3.1 Pro
- 长文 / 小说章节：implementer = Opus 4.6 1M，reviewer = GPT-5.5 + Gemini 3.1 Pro（结构 + 事实 + padding 三角）
- 方案 / 设计：implementer = Sonnet 4.6 起草，reviewer = GPT-5.5 + Opus 4.7（跨家族 + 重型 Claude 反向打）

### 3.4 冲突处理：不强求整合，两份原文都给

这条 2026-04-21 SOUL.md 拍板：**两审意见不一致时，不强求协调者整合**。协调者 agent 的"共识 / 单独 / 矛盾"三分类，本质上是把两位 reviewer 的偏见叠加上自己的偏见，最后给 Joey 一份"经过三层有损压缩"的二手判断。

**正确做法**：
- 把 reviewer A 报告原文 + reviewer B 报告原文**都直发协调者 / Joey**。
- 协调者只做**指路条**：3 行概述 + 两个文件路径，不替 Joey 拍板。
- 只有 Joey 明确说"帮我归类"才做三分类。
- 冲突项是双审最高价值的产出——单审永远产不出"冲突"——把它当成 deliverable，不是噪音。

### 3.5 求教姿态：派 reviewer 时不预设角度

派 reviewer 的 prompt 里**禁止**写"重点审 X"、"特别注意 Y"——这等于提前圈定它的视野，框架外的硬伤全漏。默认 prompt 是：

> 请全方位审，不预设角度。该坚持就坚持，该重写就重写。我们不当皇帝，是来求教的。

只有 Joey 明确指令"这次只看安全 / 只看文学性"才换。

### 3.6 实战收益举例

- **评分通胀检出**（横向校准）：单 reviewer 越审分越高很难自察；跨家族第二位横向遮 ID 重打，立刻看出标尺漂移——《靖康》卷二就是这样发现"新版甚至不如老版"。
- **padding 凑字数检出**：Claude 自己审自己的长文不会觉得 padding 异常；Gemini 一遍扫过去立刻报"§3.2 与 §3.5 几乎同段"。
- **史实错误检出**（小说项目）：Opus 写的章节自审 8 次都觉得没问题；GPT-5.5 一遍 fact-check 立刻指出"靖康年间不会有元丰改制后官职"。
- **a11y 漏抓检出**（编码项目，小虎案例）：Opus 实施 + 同家族 review，连续 3 轮都说"a11y OK"；改派 GPT-5.5 复审，立刻报出 4 个 ARIA label 缺失。

### 3.7 一句话

**1 写 + 2 审（跨家族、并发、不预设、原文直发）——这是默认配方，不是可选项。**

## 四、六顶帽子方法（角色 ≠ 分工，是思维模式）

Edward de Bono 的"六顶帽子"在多 agent 体系里被我们重新解读：**每顶帽子对应一种思维模式，用不同模型 / 不同 prompt 实例化**。

| 帽子 | 思维模式 | 在 OpenClaw 里对应什么 |
|---|---|---|
| 白帽（事实） | 只摆数据、只引证据 | Sonnet 4.6 查史料；GPT-5.5 跑真机 repro |
| 红帽（情感） | 直觉、第一反应 | Joey 自己（作者本能）；reviewer 做"读者第一感受" |
| 黑帽（批判） | 找漏洞、找风险 | Opus 4.6 抓 AI 味病灶；GPT-5.5 抓边界条件 |
| 黄帽（乐观） | 找价值、找亮点 | implementer 自评亮点；reviewer 写 strengths 段 |
| 绿帽（创意） | 替代方案、重写 | writer agent "砍不动就重写，git 兜底" |
| 蓝帽（流程） | 元层调度、判停 | 主 agent + planner agent |

**实战翻车**：早期所有 agent 都用同一种 prompt 模板（"你是专业 reviewer，请审稿"），结果产出风格、视角、措辞高度一致——三个 reviewer 跑出来像一个 reviewer 复制了三份。

**硬规矩**：
- 每顶帽子**显式命名**写进 prompt 第一段："你戴的是黑帽——你的任务是找出哪里站不住"。
- 同一轮评审**不让两顶同色帽子上场**（不要两个黑帽 reviewer，浪费一轮）。
- 蓝帽（协调者）**绝不下场戴别的帽子**——一旦主 agent 开始亲自审、亲自写，就失去了切换视角的能力。

## 五、min-weighted-scoring：防通胀的短板主导公式

这是 2026-04-22 Joey 拍板、在《靖康》项目两轮迭代出来的核心公式：

```
B 总分 = min(子项) × 0.9 + avg(子项) × 0.1
```

### 为什么不是 avg
avg 让强项兑换弱项——典型「越审越好」假象 = 优势项一直涨、短板项一直烂、平均却好看。短板必须主导。

### 为什么不是纯 min
纯取最低分会让任何一项低于 96 都不可能过 91，反向制造新通胀（reviewer 要么虚高所有项，要么集体放弃 91 标准）。

### 为什么 90/10
90% 给 min（短板真主导），10% 给 avg（4 项都不错的稍微抬一抬，避免"卡 91 必须 4 项全 96"的死板）。要更狠用 95/5、更软用 80/20，但默认 90/10 实战最稳。

### 配套四件套（缺一不可）

1. **硬扣分清单**：典型病灶项（小说里是"AI 味与作者代言控制"，代码里是"安全与边界条件"）单独立维度 + 硬扣表，每条 -5 / -8 / -10 / -15，可累加。
2. **盲式遮罩复核**：91+ 章必须显式回答"如果遮住作者名 / 轮次还能给这分吗""如果是第一稿不是修订稿还能给这分吗"，不一致 → 取较低分。
3. **双轨打分**：阶段 / 卷末 / 里程碑章天然吃情绪加分，必须问"去掉「终于到了」的满足感剩几分"，取低轨。
4. **横向校准**：每完成 5 章 / 一个阶段，遮 ID 抽样重审。早期 / 中期 / 后期各抽 1-2，发现"标尺漂移"→ 全部用最新尺重打基线。**横向校准必须跨家族（见第三章）**。

**实战翻车**：
- 旧版"min - 5"公式：第 4 项 80 分时总分卡死 75，reviewer 集体放弃努力。
- 旧版单总分：AI 味恶化被结构进步兑换掉，分数曲线一路上扬，3 卷之后才发现"越写越油"。

**硬规矩**：分数曲线**只能向上**时，第一反应是**怀疑**而不是高兴。

## 六、任务调度的硬规矩（派活铁律）

派 subagent 干活有 5 条不能破的：

1. **绝对路径**：subagent 默认 cwd 不是主 workspace，所有路径写绝对路径 `/home/zhuoyuqian/.openclaw/workspace-main/...`。曾经派任务用相对路径，subagent 跑到自己临时目录创建文件，主 workspace 找不到任何产物，回报「已落盘」。
2. **prompt 用正面表述**：写"正文用中文标点"而不是"不要 base64 / 不要英文引号"。负面警告会把动作激活进模型路径——告诉 GPT-5.5「不要 padding 凑字数」，它一半概率开始 padding。
3. **subagent 跑外部工具必 sanity check**：Sapling / GPTZero / 第三方 API 都不可信任。**贴前抓内容、贴后再抓、记录字符数 + 前后 80 字、多版本强制 A→B→C→D**。曾踩坑：subagent 报「已跑 GPTZero」，分数曲线漂亮，复查发现它 35s "读"了 35 章，根本没真跑。
4. **写完即 commit**：subagent 落盘后下一步前必 commit，不攒、不等。"等语言层扫完再说"是慢性自杀——等到出问题，根本不知道哪一步引入的。
5. **写作类不设硬超时**：`runTimeoutSeconds` 留空或给 4-8 小时，**写到自然结束**。审稿 / 校对短任务才设 30-60 分钟。

**额外两条**：
- **必读文档全列出**：v2 评分机制 + motif 全图 + 上一轮 review 报告，prompt 里给完整路径列表，不让 subagent 自己猜。
- **归档比修复重要**：reviewer 历史报告（`docs/*-review-*.md` / `reviews/*.md`）**只读**，绝不允许任何 agent 编辑或删除——包括 implementer，包括"为了清理"的好心。审稿历史是证据链。

## 七、协调者的角色定位（主 agent 不亲自做的事）

**主 agent 是协调者，不是工兵**。这条是 2026-04-09 chromium 审计任务翻车后 Joey 拍的铁律。

**唯一判断标准**：做这件事的过程中，我能立刻响应 Joey 的新消息吗？不能就委派，不管这件事看起来多小、多快。

### 禁止亲自做的事（明确清单）
- ❌ 写代码、读源码、跑 patch → spawn subagent
- ❌ **整合表格 / 多源对照表 / 综合报告** → 派 planner agent（2026-04-20 Joey 明确）
- ❌ 大段读章节、读笔记、做内容判断 → 派对应专差
- ❌ 凭印象填"已对齐 / 已修复"等状态 → 任何状态汇总都要让专差当场打开文件验证后填

### 主 agent 该做的事
- 拆解任务边界、写 task prompt
- 整合多个 subagent 结果给 Joey 听（注意：双审两份原文不整合，见 §3.4）
- 判断要不要再派一轮
- **保持对话响应**

### 上下文预算意识
- tool output 不原样转发，subagent 结果 / 文件内容 / 命令输出一律摘要后再给 Joey
- 长结果（>20 行）写文件，给 Joey 文件路径 + 3 句话摘要
- 群聊 context 共享且不可压缩，每条消息都在消耗全局预算
- 大任务必须 spawn——不光为了"协调者不下场"，也是为了不让 subagent 的工作过程污染主 session

**实战翻车**：chromium 审计那次主 agent 亲自下场读了 4 万行源码，session 被实现细节淹没，Joey 来消息时反应慢了 20 分钟，关键决策点错过窗口。

**硬规矩**：每次拿到任务前先问"这件事应该我亲自做，还是 spawn subagent？"——默认答案是后者。

## 八、Joey 和主 agent 的实战坑点（带血的教训）

| 坑 | 表现 | 加的硬规矩 |
|---|---|---|
| **评分越审越高的虚假繁荣** | 修订纵向追，分数从 78 飘到 88，结构进步真实但 AI 味在恶化 | min×0.9+avg×0.1 + AI 味硬扣 + **跨家族横向校准（§3 + §5）** |
| **subagent 偷懒装看不见** | Gemini 3.1 Pro 35s "读" 35 章，复读原文冒充 review；writer 砍 N 处但同根兄弟句不扫 | 反作弊锚点（藏特征词）+ 时间下限 + 分批 + 阅读证据文件 + 兄弟句扫荡报告必填 |
| **reviewer 预设角度变命题作文** | 派 reviewer "重点审商业流畅"，文学硬伤全漏 | 默认全方位，prompt 明写"不预设审核角度"（见 §3.5） |
| **同 family 双审 = 假双审** | 两个 Opus 互审，连续 3 轮 a11y 全漏 | reviewer 至少跨两家（见 §3.3） |
| **改 gateway 不能 ctrl-c 杀** | gateway 重启用 ctrl-c 会留残留进程 | 用第一方 `gateway` 工具的 `restart`，不用 CLI 杀法 |
| **三分类整合污染原始信号** | 协调者做"共识/单独/矛盾"分类，把 reviewer 偏见叠上自己的偏见交给 Joey | 不做三分类，两份原文直发 Joey 自己判（见 §3.4） |
| **换皮病** | 砍了 §X 对仗，§Y 用同型病换皮重新引入；漂亮化诱惑在章尾 / 卷末最强 | 改稿 prompt 里写"砍掉病灶后改写时禁止追求漂亮"+ 改完默念检测 |
| **守烂版** | 砍不动的段不敢重写，r2/r3/r4 都卡 78-79 | 改稿 prompt 里写"砍不动就重写，git 兜底" |
| **飞书消息英文引号截断** | 正文里写 `"..."` 大概率截断在引号位置 | 飞书消息禁用英文引号，用「」或反引号 |
| **长消息发 thread-reply 被吞** | >2000 字符 + 多代码块 → 静默截断，API 返 ok:true | 超长内容直接发文件，话题里只放 3-5 行概述 |
| **每步都问 Joey「要发车吗」** | 不阻塞的事被卡在"等批准" | 有清单 + 有验收 + 没歧义 = 自己派；只有阻塞决策（路线 / 笔名 / 钱）才问 |

## 九、跨项目可迁移的内核

小虎做 chromium 编码，小爪做历史小说，方法论的共性是什么？

### 三条不变的内核

1. **怀疑表面**——任何"看起来 OK"都不是证据。代码 review 的"风格 OK / 安全 OK"和小说审稿的"读起来挺顺"都是同一种表面糖衣。要的是**真证据**：上游源码 / 真机 repro / 引用带行号 / 原文带 §位置。
2. **留证据**——所有 review 报告、所有 commit、所有 subagent 输出都落盘归档。**归档 > 修复**——出问题时第一件事是回查证据链，不是猜。
3. **不信 AI 自我感觉良好**——分数曲线一路向上 = 怀疑；reviewer 全 APPROVE = 怀疑；subagent 报"已修复" = 自己再跑一遍 sanity check。模型相信自己的程度和它正确的程度无关。**唯一的破解办法是跨家族独立 reviewer（§3）**。

### 跨项目可复用的 SKILL

- `dual-review`：通用双审协议（artifact-agnostic，1 写 + 2 审标准配方的实现）
- `min-weighted-scoring`：通用防通胀打分（小说 / 代码 / 产品 / 方案 / 面试都能用）
- `dual-review-novel-loop`：小说专用闭环（writer + 双审 + 调度）

### 不可迁移的部分（要重新校准）

- 阈值（小说 81/90/95，代码可能 70/85/95）
- 硬扣清单（小说的 AI 味病灶 ≠ 代码的安全病灶）
- 子项设计（必须按当前评审对象的"典型作弊维度"重列）

## 十、一句话总结

**多模型不是为了拿多个 APPROVE 让自己安心，是为了用跨家族的独立坐标系把单 AI 的「我觉得没问题」钉在台面上；幻觉控制不是靠模型变聪明，是靠机制（1 写 + 2 审跨家族 + 实施者≠reviewer + 短板主导评分 + 协调者必自验）让它无处可逃。**

# 上下文管理与 Handoff（轻量落地版）

> 目标：避免长任务 agent（尤其读书 / 审稿 / 大规模文件分析）因为上下文积累过多而中途失忆、爆 context、重复劳动。

## 为什么要做

真实事故已经发生过：长任务 agent（如书虫）在一个 session 里读太多文件、吃了太多工具输出，却没有在阶段边界及时落盘和交接，最后上下文爆掉。

问题的根因不是“模型不够大”，而是：
- 没有 checkpoint
- 中间产出没写出到文件
- 没有明确 handoff 触发点
- 交接质量靠自觉

## 设计原则

1. **都知不碰 handoff 细节**：都知要保持 context 干净，负责路由和汇报，不负责审交接文件。
2. **handoff 质量靠模板和脚本，不靠人审**。
3. **长任务靠硬触发 checkpoint，不靠感觉**。
4. **文件是外部记忆，session 只是临时工作台**。

## 最小可行方案（MVP）

### 规则 1：Handoff 走强制模板
任何任务进入需要交接的状态（如 review / ready_to_report / waiting_user / blocked），都必须提供结构化 handoff：
- 已完成内容
- 未完成内容
- 风险
- 下一步
- artifact ref

模板见：`workspace-main/docs/handoff-template.md`

### 规则 2：长任务有 3 个硬触发 checkpoint 的时机
1. 每完成一个独立交付单元
2. session 结束前
3. 被中断恢复时（先读最新 checkpoint 再继续）

### 规则 3：不要连续跨两个工作单元只消费不产出
例如：
- 读了 5 章，就必须写 notes / checkpoint
- 审了 10 章，就必须写中间结论
- 做完一个模块分析，必须先落盘，再读下一模块

### 规则 4：脚本校验，失败一律回中书
自动校验的重点不是“写得多漂亮”，而是：
- 字段齐不齐
- artifact ref 是否可解析
- 状态迁移是否合法
- blocked / waiting_user 是否带 reason

如果校验失败，**一律中书介入修正**，不占用都知上下文。

## 各角色怎么接入

### 都知（main）
- 只看任务快照字段汇报进度
- 不读长 handoff 细节
- 不做 handoff 质检

### 中书（planner）
- 设计 handoff 字段和边界
- 校验失败后介入修正

### 门下（reviewer）
- 继续审方案和结果
- 不做 handoff 专项审核

### 书虫 / 长任务 agent
- 重点执行 checkpoint 规则
- 每个工作单元都必须落盘

## 为什么这套比“大而全”的制度更稳

因为它不把上下文管理变成新的官僚流程。

不是：
- 新增一个 handoff reviewer
- 新增一层审批
- 让都知审一堆交接文件

而是：
- 模板强制
- 脚本校验
- 长任务 agent 自己在阶段边界落盘

## 一句话总纲

> 每个节点必须留 checkpoint，每次校验由脚本执行，每次校验失败由中书收场，都知只看仪表盘，不看施工日志。
